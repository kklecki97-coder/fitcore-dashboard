Audyt i Testy w Claude Code

Krotki poradnik jak dbac o jakosc aplikacji budowanej z AI. Bez stresu, bez komplikacji.

1 Co to jest audyt?

Audyt to przeglad Twojej aplikacji pod katem problemow, ktore nie sa widoczne na pierwszy rzut oka. Claude sprawdza kod, bezpieczenstwo, wydajnosc i UX - i mowi Ci co naprawic.

Co konkretnie sprawdza:

  • Kod i architektura - czy pliki nie sa za duze, czy dane plyna dobrze, czy nie ma powtorzen
  • Wydajnosc - czy strona laduje sie szybko, czy bundle nie jest za duzy
  • Bezpieczenstwo - czy klucze API sa bezpieczne, czy inputy sa walidowane, czy ktos nie moze zhackowac
  • UX/Design - czy kolory sa spojne, czy na mobile dziala, czy jest accessible
  • Funkcjonalnosci - co dziala, co jest scaffolded (przygotowane ale nie podpiete), co brakuje

2 Prompt na full audyt

Skopiuj i wklej do Claude Code kiedy chcesz zrobic przeglad:

Zrob full audyt mojej aplikacji. Sprawdz: 1. Architektura i jakosc kodu - duze pliki, tech debt, props drilling, brak memoizacji 2. Wydajnosc - bundle size, code splitting, niepotrzebne re-rendery 3. Bezpieczenstwo - klucze API, CORS, sanitization inputow, rate limiting 4. UX/Design - spojnosc kolorow (CSS variables vs hardcoded), responsywnosc, accessibility (aria-labels, focus trap) 5. Funkcjonalnosci - co jest complete, partial, scaffolded, missing 6. Testy - jakie istnieja, czego brakuje Dla kazdego problemu podaj severity (critical/medium/low) i plik. Na koncu daj mi top 10 priorytetow posortowanych po impact/effort. NIE naprawiaj niczego - tylko raport.

3 Co to sa testy?

Testy to kod ktory automatycznie sprawdza czy Twoja aplikacja dziala poprawnie. Zamiast recznie klikac po stronie, piszesz test ktory robi to za Ciebie - i mozesz go odpalac w nieskonczonosc.

Rodzaje testow:

  • Unit testy - testuja pojedyncze funkcje (np. "czy formatPrice(100) zwraca '$100'?")
  • Component testy - testuja cale strony/komponenty (np. "czy LoginPage pokazuje formularz i loguje usera?")
  • Integration testy - testuja flow end-to-end (np. "user sie loguje, widzi dashboard, wysyla wiadomosc")

4 Prompt na napisanie testow

Napisz component testy dla mojej aplikacji. Zacznij od 3 najwazniejszych stron: 1. Strona logowania - czy renderuje formularz, waliduje puste pola, wywoluje auth 2. Glowna strona z danymi - czy wyswietla dane, czy filtry dzialaja 3. Strona z formularzem (np. tworzenie czegos) - czy modal sie otwiera, czy submit dziala Uzyj vitest + @testing-library/react. Stworz setupTests.ts z mockami (matchMedia, scrollIntoView). Stworz test-utils.tsx z wrapperem providers. Mockuj zewnetrzne serwisy (supabase, API calls). Na koncu odpal npm test i upewnij sie ze wszystko przechodzi.

5 Kiedy robic co?

Po kazdym duzym feature
Mini-audyt (10 min) - sprawdz czy nie dodales duzego tech debt
Co 2-3 miesiace
Full audyt - pelny przeglad kodu, security, wydajnosci
Przed waznym momentem
Audyt + testy - przed demo, przed pierwszym klientem, przed launchem
Na biezaco
Testy przy kazdym nowym feature - pisz test razem z kodem, nie po fakcie

6 Quick prompt na mini-audyt

Szybki przeglad po dodaniu feature - zajmuje 5 minut:

Zrob szybki przeglad ostatnich zmian. Sprawdz: - Czy nowe pliki nie sa za duze (>500 linii)? - Czy nie ma hardcoded kolorow/wartosci? - Czy nowe inputy sa walidowane? - Czy build przechodzi? - Czy istniejace testy dalej przechodza? Daj mi max 5 rzeczy do poprawy.

7 Najwazniejsze zasady

Pamietaj:

  • Audyt nie znaczy "jest zle" - znaczy "tu jest nastepny level". Kazdy projekt ma tech debt, nawet Google.
  • Nie naprawiaj wszystkiego naraz - sortuj po severity. Critical najpierw, low moze poczekac.
  • Quick wins first - szukaj rzeczy ktore zajmuja 15 min a daja duzy efekt (np. code splitting, memoizacja).
  • Testy to inwestycja - na poczatku spowalniaja, ale po 3 miesiacach oszczedzaja godziny debugowania.
  • Nie kasuj kodu bez zrozumienia - Claude podczas audytu moze zaproponowac usuniecie czegos. Zawsze pytaj "po co to bylo?" zanim usuniesz.

Typowe bledy:

  • Nigdy nie robic audytu i zbierac tech debt miesiącami - potem jest za duzo do naprawienia
  • Pisac testy po fakcie zamiast razem z kodem - trudniej i mniej motywujace
  • Ignorowac security issues bo "to mala apka" - hackuja male apki tak samo jak duze
  • Pushowac na produkcje bez npm run build - jesli build nie przechodzi, deploy sie wysypie
